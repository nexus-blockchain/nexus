# 代付功能 — 合理性与可行性分析

## 一、什么是"代付"

代付（Pay on Behalf）是指：**下单人 ≠ 付款人**。即 A 下单购买商品/服务，但由 B 来支付费用。

## 二、当前支付架构

| 环节 | 当前实现 |
|------|---------|
| **下单** | `place_order(buyer, product_id, quantity, ...)` — buyer 同时是下单人和付款人 |
| **锁款** | NEX: `Escrow::lock_from(&buyer, order_id, amount)` / Token: `EntityToken::reserve(entity_id, &buyer, amount)` |
| **结算** | Escrow → Seller + Platform |
| **退款** | Escrow → Buyer（退回下单人） |
| **Order 结构** | 只有 `buyer` 和 `seller`，没有独立的 `payer` 字段 |

**关键约束：当前 buyer 同时承担"下单人"和"付款人"两个角色，无法分离。**

## 三、合理性分析

### 3.1 业务场景 — 为什么需要代付

| 场景 | 说明 | 频率 |
|------|------|------|
| **企业采购** | 企业 Entity 为员工代付办公用品、设备 | 高频，刚需 |
| **礼品/送礼** | A 给 B 买礼物，A 付款 B 收货 | 中频 |
| **父母/监护人代付** | 未成年人或无余额用户由家长代付 | 中频 |
| **组织补贴** | 组织从公共资金池为成员代付 | 高频，与 Entity 体系天然契合 |
| **团购/集采** | 团长统一代付，分发给成员 | 中频 |
| **ShoppingBalance 不足时补差** | 用户积分不够，朋友帮补差价 | 低频 |

### 3.2 合理性结论

**合理性：高。** 理由：

1. **与 Entity 体系天然匹配** — Entity（组织）→ Shop → Product 的层级结构下，组织为成员代付是自然的业务延伸
2. **完善支付生态** — 当前系统已有 Points（积分）、ShoppingBalance、EntityToken 多种支付方式，代付是"谁来付"维度的补充
3. **增加 GMV** — 代付降低购买门槛，能显著提升平台交易量
4. **区块链场景独特需求** — 新用户可能没有足够的 NEX 代币，代付是重要的引导工具

## 四、可行性分析

### 4.1 技术方案概要

```
新增字段：Order.payer: Option<AccountId>
新增外部调用：place_order_for(payer, buyer, product_id, quantity, ...)
资金流向：payer → Escrow → Seller
退款流向：Escrow → payer（非 buyer）
```

### 4.2 需要修改的模块

| 模块 | 改动 | 复杂度 |
|------|------|--------|
| **Order 结构体** | 新增 `payer: Option<AccountId>` 字段 | 低 |
| **place_order** | 新增 `payer` 参数（`Option<AccountId>`），None 时行为不变 | 中 |
| **Escrow lock** | `lock_from(payer.unwrap_or(buyer), ...)` | 低 |
| **退款逻辑** | `refund_all(order_id, payer.unwrap_or(buyer))` | 低 |
| **cancel_order** | 退款目标改为 payer | 低 |
| **dispute/refund** | 退款目标改为 payer | 低 |
| **EntityToken reserve** | reserve 来源改为 payer | 低 |
| **Storage migration** | Order 结构变化需 migration | 中 |
| **Events** | 新增 `OrderPaidBy { order_id, buyer, payer }` 事件 | 低 |
| **Tests** | 新增代付场景测试 | 中 |

### 4.3 核心设计决策

| 决策点 | 选项 | 建议 |
|--------|------|------|
| **payer 授权** | (A) payer 主动调用 `pay_for_order` (B) buyer 下单时指定 payer，payer 需要预先授权 (C) 新增 `place_order_for` 由 payer 直接调用 | **建议 C** — 最简单，payer 自己签名即安全 |
| **退款归谁** | (A) 始终退给 payer (B) 始终退给 buyer (C) 可配置 | **建议 A** — 谁付的钱退给谁，逻辑清晰 |
| **buyer 能否取消** | (A) buyer 可取消（退给 payer）(B) 只有 payer 能取消 (C) buyer 和 payer 都能取消 | **建议 C** — 灵活性最高 |
| **payer 是否能查看订单** | (A) 只能看自己代付的 (B) 完全不能看 | **建议 A** — payer 有知情权 |
| **积分归谁** | (A) 归 buyer（使用者）(B) 归 payer（付款者）(C) 可配置 | **建议 A** — 积分奖励使用者，鼓励复购 |

### 4.4 安全风险分析

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **恶意代付（洗钱）** | payer 用非法资金为 buyer 购买，再退款洗白 | 退款始终退给 payer（而非 buyer），堵死洗钱路径 |
| **权限滥用** | 组织管理员用组织资金为自己代付 | 代付走 payer 个人账户而非 Entity 账户；Entity 代付需额外权限控制 |
| **退款争议** | buyer 和 payer 对退款归属产生分歧 | 链上规则明确：退给 payer，无歧义 |
| **重入/双花** | Escrow lock 中的并发问题 | 现有 Escrow 已有原子性保证，无额外风险 |
| **Storage 膨胀** | 每个 Order 多一个 Option<AccountId> | 极小（32 bytes per order），可忽略 |

### 4.5 与现有系统的兼容性

| 子系统 | 影响 | 兼容方案 |
|--------|------|---------|
| **Escrow** | `lock_from` 调用方从 buyer 变为 payer | Escrow 本身不关心谁是 payer，无需改动 |
| **Commission** | 平台抽佣基于订单金额 | 无影响，抽佣逻辑不变 |
| **Points** | 积分发放对象 | 仍发给 buyer，无需改 Shop 积分系统 |
| **ShoppingBalance** | 余额扣减 | 仍从 buyer 扣减（如果有），差额由 payer 补 |
| **Dispute** | 仲裁和退款 | 退款目标改为 payer |
| **Storage Migration** | Order 结构变化 | `payer: None` 兼容旧数据 |

### 4.6 实现复杂度评估

| 维度 | 评分 | 说明 |
|------|------|------|
| **代码改动量** | ⭐⭐ (小) | 核心改动 ~200 行，主要在 order pallet |
| **架构影响** | ⭐ (极小) | 不改变现有架构，纯增量添加 |
| **测试工作量** | ⭐⭐⭐ (中) | 需覆盖代付下单/退款/取消/仲裁全路径 |
| **Migration** | ⭐⭐ (小) | Order 结构新增 Option 字段，旧数据兼容 |
| **安全审计** | ⭐⭐ (小) | 资金流向清晰，无复杂逻辑 |

## 五、总结

| 维度 | 评估 |
|------|------|
| **合理性** | **高** — 与 Entity 组织体系、多元支付体系天然契合，业务场景丰富 |
| **可行性** | **高** — 技术改动小（~200行核心代码），与现有 Escrow/Commission/Points 系统完全兼容 |
| **优先级** | **建议中高** — 对 GMV 和用户增长有直接拉动，但非阻断性需求 |
| **推荐方案** | 方案 C：新增 `place_order_for(payer, buyer, ...)` 由 payer 直接签名调用 |
| **预计工作量** | Order pallet 改动 + Storage migration + 测试 ≈ 一个中等 PR |

**核心原则：谁签名谁付款，退款退给付款人，积分奖励使用者。**
