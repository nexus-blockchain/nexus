# 1. 总体判断

## 1.1 一句话结论

`pallet-entity` 不是“能力不够”，而是已经形成了一个**功能非常全、但边界尚未完全收口**的业务模块组。

它已经具备：

- 组织注册与多店铺架构
- 商品与订单全流程
- 会员、积分、购物余额、返佣
- Entity Token、发售、交易市场
- DAO 治理、披露、KYC

但当前最突出的问题不是“少模块”，而是：

1. **边界重叠**：尤其是资金域、主店控制、激励中心、KYC 接口、治理执行口。
2. **文档与代码不同步**：部分 README 仍保留旧描述，实际代码已进入下一轮重构状态。
3. **高复杂模块膨胀**：`governance`、`disclosure`、`commission`、`common/traits` 都已经接近长期维护风险区。
4. **关键闭环仍未完全收口**：尤其是店铺关闭保护、订单完成副作用链、完整售后/仲裁闭环。

## 1.2 当前最成熟的主链路

最成熟、最值得继续做强的是：

```text
registry -> shop -> product -> order -> member -> loyalty
```

这是平台最明确的商业核心闭环。

## 1.3 当前最需要收敛的复杂域

最需要瘦身和收口的是：

```text
common / governance / commission / token(激励部分) / disclosure
```

# 2. 当前模块组真实现状

## 2.1 顶层模块

当前仓库包含以下主域：

- `common`
- `registry`
- `shop`
- `product`
- `order`
- `review`
- `token`
- `market`
- `tokensale`
- `member`
- `loyalty`
- `governance`
- `disclosure`
- `kyc`
- `commission/*`

## 2.2 真实代码状态与文档状态不完全一致

当前代码已经发生了几件关键演化：

1. `loyalty` 已落地，不再只是规划中的模块。
2. `order` 内部已经拆出争议/退款子域逻辑，但还没有真正独立成顶层 `dispute`。
3. `governance` 已开始通过 `XxxGovernancePort` 执行下游治理操作，不再是全链下语义。
4. `common` 已按目录拆分，但 trait 数量仍非常多，接口仍在继续膨胀。
5. `loyalty` 代码中已经包含 **Token 购物余额**，但部分 README 仍沿用旧描述，文档存在滞后。

## 2.3 当前真实边界问题

### A. 资金域被拆散

资金职责分散在：

- `registry`：Entity 金库 / 组织资金
- `shop`：Shop 运营资金
- `commission/core`：佣金保护、提取、平台费、提现
- `loyalty`：购物余额负债

问题不在于是否必须新建 treasury pallet，而在于当前**业务语义已经分散到了多个地方**。

### B. 主店存在双控制面

与主店相关的语义同时存在于：

- `registry`
- `shop`

这容易造成：

- 事实源不唯一
- 写入口重复
- 文档解释混乱

### C. 激励中心双核心

当前激励相关能力分布在：

- `loyalty`：积分、购物余额
- `token`：reward/exchange/min_redeem 等激励参数

导致“资产规则”和“用户激励规则”混在一起。

### D. KYC 接口没有统一

当前至少有以下接口并存：

- `common::KycProvider`
- `member::KycChecker`
- `token::KycLevelProvider`
- `tokensale::KycChecker`

这是明显的接口冗余与未来一致性风险。

### E. 治理执行有真执行也有半执行

`governance` 中既有真正通过 Port 落地执行的提案，也有：

- 链下备注型
- 事件型
- 语义存在但真实执行有限的提案

治理的“提案通过”与“真实执行完成”并不完全等价。

# 3. 从不同角色视角看业务流

---

## 3.1 平台 / Root / 监管方

### 关注目标

- 全局安全
- 强制暂停/关闭
- 合规处罚
- 风险处置

### 涉及模块

- `registry`
- `shop`
- `order`
- `market`
- `governance`
- `disclosure`
- `kyc`

### 当前优点

- 已有全局 pause / ban / force close / force refund 等能力
- 监管和治理入口足够多

### 当前缺陷

- 强制操作分散在多个 pallet，缺统一审计总账
- 处罚结果没有充分向下游业务执行层联动
- 治理执行状态与链下执行回执之间缺可追踪闭环

---

## 3.2 Entity Owner / 创始人

### 关注目标

- 注册实体
- 开店
- 商品上架
- 订单成交
- 会员增长
- 积分返佣
- 发币融资
- 治理和合规

### 涉及模块

- `registry`
- `shop`
- `product`
- `member`
- `loyalty`
- `commission`
- `token`
- `tokensale`
- `governance`
- `disclosure`
- `kyc`

### 当前优点

- 基本具备从 0 到 1 的完整业务闭环
- 组织层和店铺层架构方向正确

### 当前缺陷

- 配置项太多，Owner 的认知负担过高
- 返佣模式太多，实际运营很难稳定配置
- 资本化能力过重，不适合普通商家默认使用

---

## 3.3 Entity Admin

### 关注目标

- 协助 Owner 运营
- 管控商品、订单、会员、店铺、KYC 等

### 当前优点

- 权限位足够细

### 当前缺陷

- 权限是“技术权限位”，但不是“业务岗位模板”
- 缺少预设角色模板，例如：
  - 运营管理员
  - 财务管理员
  - 合规管理员
  - 店铺管理员

# 4. 逐模块深度分析

---

## 4.1 `common`

### 当前定位

跨模块共享类型、Trait、Hook、分页和错误常量。

### 当前价值

- 是整个模块组的基础协议层
- 方向是正确的

### 当前问题

- trait 仍然很多
- 存在遗留接口
- 存在重复接口
- 一些接口已经进入“过渡层”状态

### 必须新增

- trait 生命周期分级：稳定接口 / 过渡接口 / 待移除接口
- 统一 KYC / Member 相关跨模块能力

### 冗余/应收口

- `ShoppingBalanceProvider`
- `OnOrderStatusChange`
- 各业务模块内部重复 KYC/Member trait

### 结论

**保留，但必须瘦身。**

---

## 4.2 `registry`

### 当前定位

Entity 生命周期、Owner/Admin、主店、推荐人、Entity 金库。

### 当前优点

- 组织根模型比较稳
- Entity/Shop 双层架构基础合理

### 当前问题

- 混入了较多资金语义
- 主店相关职责与 `shop` 有交叉

### 必须新增

- Entity 级经营状态总览
- 不同 EntityType 的准入规则
- 更清晰的组织级风险面板

### 冗余/应收口

- `set_primary_shop` 双入口

### 结论

**保留并增强。**

---

## 4.3 `shop`

### 当前定位

Shop 生命周期、Manager、运营资金、关闭宽限期、封禁/解封。

### 当前优点

- 关闭宽限期机制方向正确
- 日常经营能力较完整

### 当前问题

- 最终关闭前的业务依赖检查不够完整
- 与主店控制存在边界重复

### 必须新增

- finalize close 前检查：
  - 活跃订单
  - 退款/争议
  - 服务中的订单
  - 相关未平账资金
- transfer shop 的依赖校验

### 结论

**保留并增强。**

---

## 4.4 `product`

### 当前定位

商品目录、库存、押金、CID 元数据。

### 当前优点

- 边界相对清晰
- CRUD + publish/unpublish + batch 较完整

### 当前问题

- 缺商品销售快照
- 大规模 SKU 场景下押金可能过重

### 必须新增

- 订单商品快照
- SKU/变体快照
- 售后关联信息

### 结论

**保留，基本不需要大裁。**

---

## 4.5 `order`

### 当前定位

下单、支付、履约、退款、超时处理、代付、双资产支付。

### 当前优点

- 是整个系统最强的核心模块之一
- Digital / Physical / Service 三类流程区分较清晰

### 当前问题

- 完整售后域不足
- 争议更像退款子流，而不是仲裁域
- 完成/取消后的副作用链需要更强集成闭环

### 必须新增

- 完整售后/仲裁
- Token 部分退款
- 证据和仲裁结论
- 账务流水/对账
- Hook 真正生产接线

### 结论

**必须保留，并继续做强。**

---

## 4.6 `review`

### 当前定位

订单评价、商家回复、商品评分索引。

### 当前优点

- 功能完整度对轻量评价系统来说已经够高

### 当前问题

- 店铺评分和商品评分聚合存在一致性风险

### 必须新增

- shop rating delta 修正能力
- 举报/申诉机制（可后置）

### 结论

**保留，轻量增强即可。**

---

## 4.7 `member`

### 当前定位

会员注册、审批、推荐链、等级、升级规则、统计。

### 当前优点

- 业务价值很高
- 与交易和增长链路衔接自然

### 当前问题

- 单模块职责过宽
- 注册入口与 Entity 级配置之间存在 shop_id 语义绕路
- 本地 KYC trait 重复

### 必须新增

- 反作弊/反刷单
- 标签/分群
- 内部按子域拆分文件与逻辑

### 结论

**保留并内部分拆。**

---

## 4.8 `loyalty`

### 当前定位

积分 + NEX 购物余额 + Token 购物余额 + 激励入口。

### 当前优点

- 已经具备“统一忠诚度中心”雏形
- 方向正确

### 当前问题

- 文档与代码存在滞后
- 与 `token` 的激励语义仍有重叠
- `ShopShoppingTotal` 命名与 Entity 维度语义不完全一致

### 必须新增

- 统一激励控制台
- 负债视图
- 可兑付能力视图
- 到期与沉淀分析

### 结论

**保留并做成统一激励中心。**

---

## 4.9 `token`

### 当前定位

Entity Token 资产桥接、锁仓、分红、转账限制。

### 当前优点

- 资产层能力比较完整
- 已经不是简单积分模块

### 当前问题

- 与 loyalty 的激励边界不清
- 本地 KYC / member trait 重复
- 若真支持 Equity/Share/Bond，证券化能力还不完整

### 必须新增

- 真正证券化场景所需的快照和公司行动能力（若继续保留证券类 token）

### 应收口

- 本地 KYC/member trait
- 部分激励语义

### 结论

**保留，但应瘦身为资产中心。**

---

## 4.10 `tokensale`

### 当前定位

发售轮次、认购、退款、白名单、解锁。

### 当前优点

- 功能覆盖面大

### 当前问题

- 模式略多
- KYC 接口本地化
- 与 token vesting 存在语义交叉

### 必须新增

- 统一接 common 的 KYC
- 发售后联动披露/治理

### 建议收敛

优先保留：

- FixedPrice
- WhitelistAllocation
- FCFS

延后：

- DutchAuction
- Lottery

### 结论

**保留，但前期模式收敛。**

---

## 4.11 `market`

### 当前定位

Entity Token DEX / Order Book 市场。

### 当前优点

- 设计能力很强
- TWAP/熔断/KYC/内幕限制都已具备

### 当前问题

- 对普通商家过重
- 高阶订单类型过多
- 商业化和监管报表能力不足

### 建议保留

前期优先保：

- Limit
- Market
- Cancel
- Pause / Resume
- 基础 TWAP 保护

延后：

- IOC
- FOK
- PostOnly

### 结论

**保留为可选模块，但前期瘦身。**

---

## 4.12 `governance`

### 当前定位

提案、投票、执行、委托、否决、治理参数。

### 当前优点

- 有真正的治理框架能力

### 当前问题

- ProposalType 膨胀
- 存在 advisory / offchain / onchain 混用
- 并非所有提案都具备真实链上执行闭环

### 必须新增

- 提案执行分类
- 链下执行回执
- 提案收敛建模

### 结论

**保留，但必须大幅瘦身。**

---

## 4.13 `disclosure`

### 当前定位

披露、公告、内幕人、黑窗、处罚。

### 当前优点

- 合规能力很强

### 当前问题

- 复杂度已经很高
- 下游真正强消费还不够多

### 建议

默认只给资本型 Entity 开启，不应作为普通商户默认能力。

### 结论

**保留，但作为资本/强监管实体的可选能力。**

---

## 4.14 `kyc`

### 当前定位

Per-Entity KYC / Provider / 风险评分 / 有效期 / 授权体系。

### 当前优点

- 业务价值高
- 架构清晰

### 当前问题

- 下游接口未完全统一消费

### 必须新增

- 作为全系统统一 KYC 能力源
- sanctions / AML / PEP 等增强能力（中长期）

### 结论

**保留并统一全系统入口。**

---

## 4.15 `commission/*`

### 当前优点

- 模式丰富
- 已形成完整引擎能力

### 当前问题

- 模式重叠严重
- 运营认知负担极高
- 合规/风控复杂

### 分项结论

#### `commission/core`
保留，但作为结算内核，不能继续膨胀为“万能运营控制台”。

#### `referral`
强保留，最通用、最易理解。

#### `multi-level`
保留，但只作为高级版，不建议默认启用。

#### `team`
保留，但必须统一团队业绩口径。

#### `level-diff`
建议弱化，不作为主推模式。

#### `single-line`
建议冻结/非默认/高风险能力。

#### `pool-reward`
建议可选保留，不作为普通商业主链路刚需。

# 5. 必须新增的功能（P0/P1）

## 5.1 P0：必须马上补

### P0-1 店铺关闭最终保护

在 `finalize_close_shop` 前强制检查：

- 活跃订单
- 活跃退款/争议
- 服务类未完成订单
- 相关未平账负债

### P0-2 订单完成/取消 Hook 正式接线

必须至少接通：

- `commission`
- `member`
- `loyalty`
- `shop stats`

### P0-3 售后升级为完整争议域

补齐：

- 证据
- 仲裁
- 部分退款
- Token 部分退款
- 退货/换货/RMA

### P0-4 KYC 接口统一

统一到：

- `common::KycProvider`

### P0-5 治理提案分级

所有提案必须明确：

- `OnChainExecutable`
- `OffChainRequired`
- `AdvisoryOnly`

### P0-6 评价评分一致性修复

增加：

- 店铺评分差量更新能力

# 9. 最终目标架构建议

## 9.1 商业基础版（默认）

```text
registry
  -> shop
      -> product
      -> order
          -> member
          -> loyalty
      -> review
```

适用于绝大多数 Merchant / Service Provider。

```text
基础版
  + commission/core
  + referral
  + multi-level
  + team
```

不建议默认启用：

- `single-line`
- `level-diff`

```text
基础版
  + token
  + tokensale
  + market
  + governance
  + kyc
  + disclosure
```

仅适用于：

- Project
- DAO
- Fund
- Enterprise

不建议对普通商户默认开放。
